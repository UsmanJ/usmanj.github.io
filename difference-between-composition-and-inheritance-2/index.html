Difference between 'Composition' and 'Inheritance'

Inheritance is when a class inherits fields and methods from its superclass. However subclasses are able to override these in order to change the default behaviour.

An example of this is presented below:

Class Ships

def initialize
@hitcounter = 0
end

def hit
@hitcounter += 1
"Sunk!" if @hit_counter >= 1
end
end

Class Submarine < Ships

End

sub = Submarine.new
sub.hit
"Sunk!"

Notice in the above example we did not have to create a 'hit' method in the 'Submarine' class as we inherited it from the 'Ships' class. However, the Submarine 'Sunk!' after 1 hit. Submarine should be hit 5 times for it to sink. For this we would have to override the inherited method. This is displayed below:

Class Ships

def hit
@hitcounter += 1
"Sunk!" if @hit_counter >= 1
end
end

Class Submarine < Ships
def hit
@hitcounter += 1
"Sunk!" if @hit_counter >= 5
end
End

sub = Submarine.new
sub.hit
sub.hit
sub.hit
sub.hit
sub.hit
"Sunk!"

In the above example Battleship 'is-a' Ship. If we used composition rather than inheritance we could let a battleship use functionality from Ship, without having to BE a ship. To do this we could use a module which contained the key 'ship-like' behaviour.

We may prefer composition to inheritance in situations in which we need a little more flexibility, perhaps when we are unsure as to how future classes will behave. The danger in using inheritance is that the sub-class is reliant on the superclass, so anything changed in the super-class will have knock on effects to all its sub-classes. This could turn the minor process of making small tweaks into a much longer and time consuming process.
