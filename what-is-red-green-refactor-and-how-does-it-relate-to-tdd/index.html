What is Red-Green-Refactor and how does it relate to TDD?

So what is Red-Green-Refactor I hear you asking?

Red-Green-Refactor is one of many methods for conducting TDD (test driven development). The steps are red, green and refactor. These are defined below:

Think - Break down your problem into smaller parts so that a test can be written for each part.

Red - Write the test for your first problem. This should be short and concise. The test should fail as you will not have any code written for it to pass at this stage.

Green - Write some code, just enough; so that the test that you just wrote passes.

Refactor - Now that your test has passed, go back and refactor your code to make it clearer and more readable. Try to get rid of any duplications and excess code. Run the test again to ensure it still passes.

Repeat - This can be repeated a number of times until you are satisfied that your code is readable, passing the tests and concise.

The first working version of any code can always be improved - even if its not messy. Therefore you would use Red-Green-Refactor to improve the code, make it cleaner, make it more readable and also remove any duplications. Using this method allows you to refactor safely without breaking any code as you are constantly testing the code.
